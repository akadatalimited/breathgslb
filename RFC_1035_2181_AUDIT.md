# RFC 1035 / 2181 Clause Audit

This document is the detailed answer-path audit behind
[RFC_COMPLIANCE.md](RFC_COMPLIANCE.md) for the two core authoritative DNS
standards BreathGSLB depends on most heavily:

- [RFC 1035](https://www.rfc-editor.org/rfc/rfc1035.html)
- [RFC 2181](https://www.rfc-editor.org/rfc/rfc2181.html)

It is intentionally scoped to BreathGSLB's implemented authoritative feature
surface, not every possible DNS server feature.

## Status Meanings

- `Pass`: implemented and covered by current code/tests
- `Partial`: materially implemented, but with a documented boundary or missing
  corner-case coverage
- `Out of scope`: not implemented by design in the current product

## Important Boundaries

### `max_records`

`max_records` remains a product feature rather than a pure RFC behavior. With
the default `max_records: 0`, BreathGSLB returns full RRsets and relies on
transport truncation when needed. If an operator sets `max_records` to a
positive value, that intentionally caps returned address records and should be
treated as a product-level policy override rather than a strict RFC 2181
behavior.

### General delegation/referral behavior

BreathGSLB is now much closer to a correct authoritative-only server, but this
audit does not claim complete referral/zone-cut behavior for every possible
multi-zone parent/child arrangement. That remains a `Partial` area until it is
tested explicitly.
